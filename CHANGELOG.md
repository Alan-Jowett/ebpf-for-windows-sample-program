## Release Notes Format

Each release entry should include:

### Added
- New features, sample programs, or capabilities

### Changed  
- Changes to existing functionality or documentation

### Deprecated
- Features or samples that will be removed in future versions

### Removed
- Features or samples that have been removed

### Fixed
- Bug fixes and corrections

### Security
- Security-related improvements or fixes

## Version Numbering

This project uses [Semantic Versioning](https://semver.org/):

- **MAJOR.MINOR.PATCH** (e.g., 1.2.3)
- **MAJOR**: Breaking changes to sample structure or APIs
- **MINOR**: New sample programs or significant enhancements  
- **PATCH**: Bug fixes and minor improvements

## Release Process

1. Update this CHANGELOG.md with all changes since the last release
2. Update version numbers in relevant files
3. Create a release tag following the versioning scheme
4. Publish release notes on GitHub
5. Update documentation as needed
